# Serverpod Latest Version and Full-Stack Architecture Best Practices (v2.x / v3.0 Industrial)

## Goal
Implements End-to-end Type Safety and code generation using Serverpod. [Serverpod](https://serverpod.dev/) is an open-source Dart backend framework designed specifically for Flutter. Its greatest strength lies in establishing a seamless, strongly-typed connection from the database (PostgreSQL), through the backend API, down to the Flutter frontend.

Currently, the latest major release is Serverpod 3 (codenamed Industrial), which introduces a brand new Web Server and a refactored Auth module.

## Instructions

### Core Concept: Why Serverpod?
In traditional development, Flutter frontends and Node.js/Go backends must maintain contracts via Swagger or manually written JSON, which is highly error-prone.
Serverpod's philosophy:
1. **Write a single YAML protocol**: Serverpod automatically generates the database tables, the backend ORM models, and the frontend API Client.
2. **Share a single language (Dart)**: You can even place shared business logic in a `shared` directory to be called by both ends.

### Deconstructing the Project Architecture
When you create a project via `serverpod create myapp`, you get three modules:

1.  **`myapp_server`**: Your Dart backend server (containing Endpoints and Database connection configs).
2.  **`myapp_client`**: **NEVER MANUALLY MODIFY THIS FOLDER**. This is the frontend API Client automatically generated by Serverpod.
3.  **`myapp_flutter`**: Your Flutter App, which already automatically depends on `myapp_client`.

### Defining Data and API Contracts (Protocol)
In `myapp_server/lib/src/protocol/`, define your database and API Models by writing YAML.

**Example: Defining a User model (`user.yaml`)**
```yaml
class: User
table: user # Specifying a table name means it will be stored in the database
fields:
  name: String
  email: String
  age: int? # nullable
  createdAt: DateTime
```

After modifying the YAML, **you must** enter the `myapp_server` folder and execute:
```bash
serverpod generate
```
This command generates the backend Dart Model and the Client classes used for frontend calls.

### Backend Endpoints Development
Write business logic in `myapp_server/lib/src/endpoints/`.

*   **Endpoint classes** must inherit from `Endpoint`.
*   **The first parameter of any method** is always `Session session`, which contains database connections, Logs, and Auth information.

```dart
import 'package:serverpod/serverpod.dart';
import '../protocol/protocol.dart';

class UserEndpoint extends Endpoint {
  
  // 1. Insert data (via generated ORM)
  Future<User> createUser(Session session, User user) async {
    // 🌟 Best Practice: Handle business logic before saving
    user.createdAt = DateTime.now();
    
    // insertRow automatically writes data to PostgreSQL and returns the object containing the ID
    return await User.db.insertRow(session, user);
  }

  // 2. Query data
  Future<List<User>> getAdultUsers(Session session) async {
    return await User.db.find(
      session,
      // 🌟 The power of Serverpod: Fully strongly-typed WHERE clause queries. No more raw SQL string concatenation.
      where: (t) => t.age > 18,
      orderBy: (t) => t.createdAt,
      orderDescending: true,
    );
  }
}
```
After writing the Endpoint, run `serverpod generate` again. Your Flutter Client will instantly gain a `client.user.createUser()` method!

### Flutter Frontend Integration Architecture (MVVM / State Management)
On the Flutter side, the Serverpod Client is usually a Singleton or declared globally.
**Best Practice: Combine with Riverpod to provide dependency injection and asynchronous state management.**

#### 5.1 Global Client Provider
```dart
import 'package:myapp_client/myapp_client.dart';
import 'package:serverpod_flutter/serverpod_flutter.dart';
import 'package:flutter_riverpod/flutter_riverpod.dart';

// Setup Serverpod Client
final client = Client('http://localhost:8080/')
  ..connectivityMonitor = FlutterConnectivityMonitor();

// Declare as a Provider for dependency injection anywhere
final serverpodClientProvider = Provider<Client>((ref) => client);
```

#### 5.2 Combining with Riverpod AsyncNotifier
Use `FutureProvider` or `AsyncNotifierProvider` to call backend APIs. This elegantly handles Loading and Error states.

```dart
@riverpod
class UsersList extends _$UsersList {
  @override
  Future<List<User>> build() async {
    final client = ref.watch(serverpodClientProvider);
    // Call the backend API just like a local function
    return await client.user.getAdultUsers(); 
  }

  Future<void> addUser(User newUser) async {
    final client = ref.read(serverpodClientProvider);
    state = const AsyncValue.loading();
    state = await AsyncValue.guard(() async {
      await client.user.createUser(newUser);
      return await client.user.getAdultUsers(); // Re-fetch to refresh
    });
  }
}
```

### Advanced Serverpod Best Practices

1. **Database Migrations**:
   Every time you modify the Protocol YAML (when the `table` structure changes), you must generate a Migration script:
   `serverpod create-migration`
   And execute it during deployment or server startup to upgrade PostgreSQL.
2. **Authentication System (Auth)**:
   Never reinvent the wheel! It is strongly recommended to use the official `serverpod_auth` module. It features built-in Email/Password, Google, and Apple social logins, and the frontend UI is already packaged (`serverpod_auth_email_flutter`, etc.).
3. **Caching**:
   If some data is read extremely frequently and rarely changes, use the Session's built-in Redis cache:
   `await session.caches.local.put('my_key', myData);`
4. **Exceptions**:
   Don't let the backend throw unknown HttpExceptions directly. Define an `Exception` type in YAML and `throw` it in the Endpoint. The Flutter side can precisely catch it using `try-catch` and provide appropriate prompts.

## Constraints
* Utilize Serverpod strictly for full-stack Dart projects ensuring the protocol YAML is the sole source of truth for both sides.
* Make sure `serverpod generate` is run after every change to `.yaml` files or Endpoint files.
