# Serverpod Core Architecture and Data Models (Core & Models)

## Goal
Implements Serverpod's "Data structure as contract" core philosophy. By reading `.yaml` files in the root directory, Serverpod automatically generates full-stack, Type-safe Dart code (including Server-side ORM and Client-side API).

## Instructions

### Understanding Project Structure
A standard Serverpod project contains three main directories:
1. **`<project>_server`**: Where backend logic and Endpoints reside.
2. **`<project>_client`**: ⚠️ **NEVER MANUALLY MODIFY**. This is the Client API interface code generated by the framework based on the Server.
3. **`<project>_flutter`** (or other frontend): Your frontend project. You only need to import the Client package and initialize it here.

### Defining Data Models and Relations
Under `<project>_server/lib/src/models/`, create `.yaml` files to define data structures. **Serverpod generates not only object properties but also PostgreSQL Migrations and a powerful ORM.**

#### 2.1 Basic Model
```yaml
# user_profile.spy.yaml
class: UserProfile
table: user_profile
fields:
  name: String
  email: String
  age: int? # Adding ? means nullable
  joinedDate: DateTime
```

#### 2.2 Relations Best Practices
Serverpod v2.0+ supports powerful "Native Relations".

**One-to-Many (1:N)**: Suppose a Company has multiple Employees.
```yaml
# company.spy.yaml
class: Company
table: company
fields:
  name: String
  # 🌟 Adding the 'relation' keyword generates a list of objects in Dart
  employees: List<Employee>? relation
```

```yaml
# employee.spy.yaml
class: Employee
table: employee
fields:
  name: String
  # 🌟 Establish the relation here. Because it is the "Many" side,
  # Serverpod automatically generates a Foreign Key named 'companyId' in the database.
  company: Company? relation
```

**Many-to-Many (M:N)**: Cannot be defined directly. You must manually create a **Junction Table / Model** serving as a bridge, just like in a traditional relational database.
For example, Students and Courses:
```yaml
# enrollment.spy.yaml (Serves as the bridge)
class: Enrollment
table: enrollment
fields:
  student: Student? relation
  course: Course? relation
```

#### 2.3 Inheritance and Polymorphism
If multiple objects share fields, Serverpod supports class inheritance. This is extremely useful for Endpoint design.
```yaml
class: Animal # Parent class
fields:
  name: String
---
class: Dog # Child class
extends: Animal # 🌟 Inherits from Animal
fields:
  barkVolume: int
```
> **Advantage**: In an Endpoint, you can design an API that receives an `Animal` type, and then generously pass a `Dog` object from the Client side. Serverpod will automatically handle serialization and deserialization.

### Generating and Deploying Models
After writing the YAML, **always remember to execute**:
```bash
serverpod generate
```
If `table:` is specified (meaning a physical table should be created in the DB), you must create a Migration:
```bash
serverpod create-migration
```

## Constraints
* Always treat the configuration files in `<project>_server/lib/src/models/*.yaml` as the supreme source of truth for the database schema.
* Do not bypass Serverpod's migration scripts; always use `serverpod create-migration` when making schema changes.
